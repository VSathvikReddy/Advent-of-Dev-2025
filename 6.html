Today let's explore a few rather basic stuff which are overlooked upon.

static

static is a keyword that has multiple different use cases. Let's look at each of them

static inside a function 

int counter() {
    static int x = 0;
    x++;
    return x;
}

int main() {
    counter(); // returns 1
    counter(); // returns 2
}
  • Initialized once
  • Value persists across calls
  • Lives for the entire program
  • But scope is still local to that function

static global variables

  • Usually a vairable defined in the global scope is visible across files.
  • This could potentailly cause naming issues across files!!
  • To avoid this, we use the keyword static
#include <iostream>
using namespace std;

int x = 0; // avaiable for external linkage
static int y = 1; // not avaiable for external linkage

Here, external linkage refers to the usage of the keyword extern which is used to "find" the variable across the global scopes of other files.

An Overview:

  • Global static variables have internal linkage -> visible only in this translation unit (file).
  • Normal globals have external linkage -> visible across multiple files via extern.

static class attributes 

class A{
public:
    static int hehe;
};

int A::hehe = 0; // this is necessary, else we face a *linker* error

int main() {
    A::hehe = 2; // valid access
    A a;
    a.hehe = 15; // also a valid access
}
  • These are attributes of a class/struct that are of type static <type>.
  • These attributes have to be initialised in the global scope, otherwise we would face linked errors.

static class methods 

class A {
public:
    int attr;
    static int calc(int a, int b) {
        // this->attr = 0; // error!!
        return a + b;
    }
};

int main() {
    A a;
    A::calc(1,2); // valid
    a.calc(a.attr,a.attr); // also valid
}
  • Does not have a this pointer
  • Can be called without an object using <class_name>::<static_function_name>(param_list)
  • Cannot access non-static members, can be seen as consequence of not have access to a this pointer. I hope you guys see why it does no make sense to let it use a this pointer.

namespace

You all would definitely have using the statement using namespace std; but do you know what that actually means?

First lets explore what a namespace is.

A namespace in C++ is simply a named scope that groups together identifiers (variables, functions, types, templates) to prevent name collisions and control lookup.

There are multiple other features on top of namespaces, but this fundamental idea does not change.

#include <iostream>

namespace let_that {
    int sink_in (int a) {
        if(a < 10) {
            printf("let that sink in :)\n");
            return sinks[a];
        }
        else {
            printf(":(\n");
            return -1;
        }
    }
    int sinks[10];
}

int main() {
    let_that::sinks[4] = 20;
    int k = let_that::sink_in(4);
}

Please do note that the order of definition inside the namespace matters.

Having the array sinks after the function would result in a compilation error.

namespace game { void start(); }

// Much later, or in another file:
namespace game { void stop(); }

Both the game namespaces above would refer to the same namespace.

Namespaces are open and can be extended at any point of the file or even in different files.

The namespace std is extended in every standard library you include.

Argument Dependent Lookup (ADL)

ADL says that if a function is called with an argument of type T, search T’s namespace for matching functions.

#include <iostream>

namespace N {
    struct A {};
    void f(A) { std::cout << "N::f\n"; }
}

int main() {
    N::A a;
    f(a);  // ADL finds N::f even without "using namespace N"
}

Don't worry we'll come to using soon :)

Anonymous Namespace

This basically gives the same effect as the static keyword for internal linkages.

#include <iostream>

namespace {
    int helo = 0;
}

int main() {
    printf("%d\n",helo); // prints 0 as expected
}

Nesting namespaces

Namespaces can be nested :)

namespace A {
    namespace B {
        int helo; // accessed as A::B::helo
    }
}

Inline namespaces

This feature gives us the notion of versioning

#include <iostream>
namespace A {
    namespace v1 {
        int helo = 0;
    }
    namespace v2 {
        int helo = 1;
    }
    inline namespace v3 {
        int helo = 2;
    }
}

int main() {
    printf("%d\n",A::helo); // this prints A::v3::helo
}

Fun fact: If I have contradicting inlines, basically same identifier in two inlined namespaces of the same namespace, you would expect an error, but you would not have an error till you access this ambiguous value.

using

Time to address the elephant(?? maybe ig) in the room.

I'm sure this is the keyword most people just blindly use without know what it even does.

So let's look at an example ;)

#include <iostream>
using namespace std;

int main() {
    cout << "hehe\n";
}

As we all would know, with the using namespace std;, this file will not compile.

After our (newly gained??) knowledge of namespaces this using term would make more sense.

So basically the using namespace std; just tells the compiler, "For the rest of this scope, if you see an unqualified name, also look inside this namespace."

A few common misconceptions:

  1. using namespace X; does not bring any symbols into the scope. It simply adds another place for unqualified lookups.
  2. It applies only to the current scope.
void foo() {
    using namespace std;
    cout << "foo\n"; // works
}

int main() {
    cout << "hello\n"; // invalid, causes an error
}
  1. It can cause name conflicts, say two namespaces you included have foo. If I am "using" both the namespaces then an unqualified foo would be ambiguous.
  2. using namespace X; causes name pollution, you might recieve names that you will not be using in your program. There is a fix to this which will be dicussed after this.
  3. Be careful while using this in header files, rather try avoiding it all together. Since headers are included in multiple places, this can cause issues.
  4. using namespace std; puts every name in std into the lookup set, which slows ADL-heavy code.

There is one more use case for the keyword using. You can use it as follows

#include <iostream>
using std::cout;
using std::endl;

int main() {
    int n;
    cin >> n; // will still cause issues
    cout << "Hello World!" << endl; // works fine
}

You can use specific identifiers from a namespace as shown above.

This reduces name pollution significantly.

Now you can start naming your variables max, min and all :)

const

In my last AoD, we saw constexpr and consteval.

Now we see a more commonly used keyword from the same family called const.

const does not make the underlying object immutable.
It only prevents modification through that expression.
Other references/pointers may still modify it.

int main() {
    int x = 1;
    const int* p = &x;
    x = 2; // this is fine
    *p = 2; // this is not fine, will cause compilation error.
}

const on pointers 

int* p; // pointer to an int
const int* p; /*(or)*/ int const* p; // pointer to a const int
int* const p // constant pointer to an int
const int* const p // constant pointer to a const int

const in methods/functions 

const T foo() {} // function returns a type const T
T foo() const {} // this is only applicable for non static methods
                 // no attribute of the object is supposed to change
                 // with the exception of mutable attributes.

const family of keywords

const -> value can't be changed through the related name

constexpr -> can be evaluated at compile time (slightly different for functions and variables but yea we know :))

consteval -> function has to be evaluated at compile time

constinit -> New guy in the AoD town. Must be initialised in compile time, but may become a runtime constant later. You can read about it but we won't be focusing on it.

#include <iostream>
#include <string>
#include <cassert>
#include <expected>

using std::cout;
using std::string;
using std::endl;
using std::expected;
using std::unexpected;

namespace A{
    namespace v1{
        int counter = 0;
    }
    inline namespace v2{
        int counter = 0;
    }
}

expected<int,int> old_div_fn(int a, int b) {
    A::v1::counter++;
    if(b == 0) return A::v1::counter;
    return a/b;
}

expected<int,int> div_fn(int a, int b) {
    A::counter++;
    auto val = old_div_fn(a,b);
    if(val && b != 0) {
        return *val;
    }
    else if(val) {
        return *val - A::counter;
    }
    else {
        return unexpected(val.error() + A::counter);
    }
}

int main() {
    auto result = div_fn(3,2);
    if(!result) {
        cout << "Bertolt " << result.error() << endl;
    }
    else {
        cout << "Bertholdt " << *result << endl;
    }
    result = old_div_fn(2,0);
    if(!result) {
        cout << "Burrito " << result.error() << endl;
    }
    else {
        cout << "Boruto " << *result << endl;
    }
    result = div_fn(14,0);
    if(!result) {
        cout << "Burntoast " << result.error() << endl;
    }
    else {
        cout << "Bee Swarm Simulator "<< *result << endl;
    }
}

(a) Will the given program compile or not? If it does, What's the output.
(b) Give reasoning for your answer.

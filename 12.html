PImpl - Pointer To Implementation

The Problem

Say we are building a Renderer for a Game Engine. This class needs to talk to the GPU using API's like DirectX or Vulkan. Abstracting away the details, one would write it something like this. The issue lies in the fact that we are forced to include the massive headers for these API's.

// Renderer.hpp 
#pragma once
// We are forced to include these massive headers here
#include <vulkan/vulkan.h> 
#include <d3d12.h>
#include <vector>

class Renderer {
public:
    void drawFrame();
private:
    VkInstance vulkanInstance;      // Vulkan type
    ID3D12Device* dxDevice;         // DirectX type
};

The Consequences?

  • Dependency Pollution: Any file that includes #include "Renderer.hpp" inadvertently pulls in thousands of lines of Vulkan and DirectX code. This causes namespace pollution and macro conflicts.

  • Increased Compilation Time: If you add/modify a private member in the Renderer.hpp, the timestamp of Renderer.hpp changes. Thus every single file that includes it must be recompiled and hence the compilation time increases exponentially.

  • ABI Instability : If you ship this engine as a DLL, changing the size of Renderer.hpp breaks the binary compatability. The user must recompile their game code to match the new object size.

The Solution

Like Wheeler quoted "Any problem in computer science can be solved with another layer of indirection", we add an indirection to the Renderer.hpp

#pragma once
#include <memory> 

class Renderer {
public:
    Renderer();
    ~Renderer(); // Declaration only

    // Move semantics, PImpl is move-only by default
    Renderer(Renderer&&) noexcept;
    Renderer& operator=(Renderer&&) noexcept;

    void drawFrame();

private:
    // Forward declaration
    struct EngineInternal; 

    // The Opaque pointer which has a fixed size
    std::unique_ptr<EngineInternal> m_pImpl;
};

All the dirty details lie in Renderer.cpp

#include "Renderer.hpp"

// Heavy dependencies are isolated to this one Translation Unit
#include <vulkan/vulkan.h>
#include <d3d12.h>
#include <iostream>

// The Actual Definition
struct Renderer::EngineInternal {
    VkInstance vulkanInstance;
    ID3D12Device* dxDevice;
    
    void backendDraw() {
        // ... some logic to interact with the gpu
    }
};

// Lifecycle Implementation 

// Constructor, allocates the implementation on the heap
Renderer::Renderer() 
    : m_pImpl(std::make_unique<EngineInternal>()) {}

// Destructor, defined here, where EngineInternal is fully defined
Renderer::~Renderer() = default;

// Move Operations
Renderer::Renderer(Renderer&&) noexcept = default;
Renderer& Renderer::operator=(Renderer&&) noexcept = default;

// Interface Forwarding
void Renderer::drawFrame() {
    m_pImpl->backendDraw();
}

Why does it work?

To understand why this works, we must look at the C++ build process

  1. Memory layout and offsets

    • When the compiler compiles a client file, it needs to know how much memory to allocate for Renderer
      • Without PImpl, Size = sizeof(VkInstance) + sizeof(ID3D12Device*) + .... If implementation changes, size changes.
      • With PImpl, Size = sizeof(std::unique_ptr), a constant.
    • Because the size is a constant, the memory layout of the Renderer (ideally) never changes, even if you rewrite the entire rendering backend.

  2. The Build System

    • Say some other file, here LevelEditor.cpp uses Renderer.hpp, Renderer.cpp + vulkan.h becomes one huge translation unit, LevelEdtor.cpp and Renderer.hpp become another translation unit. They are "separate" entities.
    • Now let's just say for the sake of argument, we modify Renderer.cpp to optimize the backendDraw(). Renderer.cpp gets a new timestamp, the build system recompiles the Renderer.cpp into Renderer.o and thus the header Renderer.hpp is untouched, this means that the LevelEditor.cpp is never recompiled due to its timestamp being not modified.
    • The linker simply links the new Renderer.o and the existing LevelEditor.o.

Question 3 : A Leap of faith

This code will cause compilation error, identify all lines which can cause compilation error.

  • DataProcessor.hpp

    #pragma once
#include <memory>
#include <string>

class DataProcessor {
public:
    DataProcessor();
    ~DataProcessor(); 

    void process(const std::string& input);
    int getProcessedCount() const;

private:
    struct Impl;
    std::unique_ptr<Impl> m_pImpl;
};

  • DataProcessor.cpp

    #include "DataProcessor.hpp"
#include <algorithm> 

struct DataProcessor::Impl {
    int executionCount = 0;
    std::string lastError;
    
    void runCoreLogic() { 
        executionCount++; 
    }
};

DataProcessor::DataProcessor() : m_pImpl(std::make_unique<Impl>()) {}
DataProcessor::~DataProcessor() = default;

void DataProcessor::process(const std::string& input) {
    m_pImpl->runCoreLogic();
}

int DataProcessor::getProcessedCount() const {
    m_pImpl->executionCount++; 
    return m_pImpl->executionCount;
}

  • main.cpp

    #include "DataProcessor.hpp"

int main() {
    DataProcessor dp1;
    dp1.process("data");
    
    DataProcessor dp2 = std::move(dp1);

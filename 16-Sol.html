1. Getting a bit curious

In CRTP, we do not use inheritance for Type Erasure (treating different things as the same). We use inheritance for Functionality Injection (giving different things the same skills).

Here are a few reasons why we would use inheritance in CRTP, even without heterogeneous containers.

  1. Free behaviour : Use the Base class to inject "free" behavior into the Derived class based on a small set of primitives provided by the Derived class.

    Imagine you are building a math library. You have a Vector2, Vector3, and Matrix4x4. You want all of them to have !=, >, <=, and >= operators.

    Without Inheritance, you have to manually write all 6 comparison operators for every single class. With CRTP Inheritance, you write them once in the Base.

    template <typename Derived>
struct Comparable {
    // The Base class defines complex logic using the Derived class's simple logic
    friend bool operator!=(const Derived& lhs, const Derived& rhs) {
        return !(lhs == rhs); // Uses Derived's ==
    }
    friend bool operator>(const Derived& lhs, const Derived& rhs) {
        return rhs < lhs;     // Uses Derived's <
    }
    friend bool operator<=(const Derived& lhs, const Derived& rhs) {
        return !(lhs > rhs);
    }
    // ... and so on
};

// Derived class only does the bare minimum: implement == and <
class MyInt : public Comparable<MyInt> {
public:
    int value;
    bool operator==(const MyInt& other) const { return value == other.value; }
    bool operator<(const MyInt& other) const { return value < other.value; }
};

int main() {
    MyInt a{10}, b{20};
    
    // Got this for from the Base class!
    if (a != b) { ... } 
}

  2. Specialized Control : Sometimes you have a complex algorithm where almost all of the code is identical across all classes, but some specific section is different. You don't want to copy-paste code into every class. You want the Base class to hold the skeleton of the algorithm and call the Derived class for the specific steps.

    Example : Every parser needs to open a file, read bytes, handle errors, and close the file. The only thing that changes is how it interprets the text.

    template <typename Derived>
class GenericParser {
public:
    void parseFile(const char* filename) {
        open(filename);
        if (error) {
            handleError();
        }
        static_cast<Derived*>(this)->parseLine(currentLine);
        close();
    }
};

class JsonParser : public GenericParser<JsonParser> {
public:
    void parseLine(string line) { /* JSON specific logic */ }
};

class CsvParser : public GenericParser<CsvParser> {
public:
    void parseLine(string line) { /* CSV specific logic */ }
};

2. Answer: Undefined Behavior

Why: When you delete a derived object through a pointer to a base class, the base class must have a virtual destructor. If it doesn't, the C++ standard states this is Undefined Behavior. Typically, the Derived destructor is never called, leading to memory leaks.

Fix: Either make the Base destructor virtual (which adds vtable overhead, defeating the point of CRTP) or make the Base destructor protected (preventing deletion via base pointer). But it is not a standard practice as far as I have seen to use pointers this way in the CRTP idiom so we dont really have to worry about such cases.

1. Destructor in Header

It will fail to compile.

std::unique_ptr needs to generate code to delete the implementation object. To do this safely, the compiler must see the full definition of struct Impl to verify its size and invoke its destructor. In the header, Impl is only forward-declared, an incomplete type, so the compiler throws an error.

2. Copying without Custom Constructor

It will fail to compile.

std::unique_ptr is move-only; its copy constructor is deleted. Because your class contains a non-copyable member, the compiler implicitly deletes the copy constructor of your class. To support copying, you must write a custom copy constructor that manually clones i.e. deep copies the Impl object.

3. Lines causing compilation error

  1. main.cpp: DataProcessor dp2 = std::move(dp1);

    Reason :

    • You have explicitly declared a destructor ~DataProcessor(); in the header. In C++, declaring a user-defined destructor suppresses the automatic generation of the Move Constructor. The compiler does not generate one for you.
    • When the move constructor is missing, std::move attempts to fall back to the Copy Constructor const DataProcessor&. However, because your class contains a std::unique_ptr which is non-copyable, the implicit Copy Constructor is deleted.

    Fix :

    // In .hpp
DataProcessor(DataProcessor&&) noexcept;

// In .cpp
DataProcessor::DataProcessor(DataProcessor&&) noexcept = default;

  2. main.cpp: DataProcessor dp3 = dp2;

    Reason:

    • std::unique_ptr has its copy constructor deleted it cannot be copied, only moved.

    • Because DataProcessor holds a std::unique_ptr member m_pImpl, the DataProcessor class itself becomes non-copyable. The compiler implicitly deletes the copy constructor for DataProcessor.

    Fix: Implement a custom Copy Constructor that allocates a new Impl and copies the values manually.

  • NOTE: Not a Compilation Error

    You might have suspected DataProcessor.cpp lines modifying executionCount inside a const function would be an error.

    int DataProcessor::getProcessedCount() const {
    m_pImpl->executionCount++; 
    return m_pImpl->executionCount;
}
    • Why it compiles: This is a quirk of PImpl with smart pointers known as "Physical Constness" vs. "Logical Constness."
      • The const on the function makes m_pImpl a const std::unique_ptr.
      • This means you cannot change the pointer. However, std::unique_ptr does not propagate constness to the object it points to. m_pImpl-> returns a non-const Impl*. Therefore, modifying executionCount is technically allowed by the compiler, even though it likely violates your design intent.

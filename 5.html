Views

In any kind of programs, copying data more than required is the common enemy of performance. Every malloc, memcpy and atomic reference count operations decreases the overall throughput and pollutes the CPU cache.

Modern C++ solves this with views, a non-owning lightweight objects that refer to existing data. Often called Fat Pointers because they typically just consist of two registers of data :

  1. A pointer
  2. A size

Some of the more commonly used views in C++ are :

1. std::string_view

Introduced in C++ 17, std::string_view (found in <string_view> header) is a light-weight non-owning reference to a character sequence. While a std::string is a complex object that manages memory (heap allocation, size, capacity), a std::string_view is essentially a tiny struct containing just two pieces of data.

class string_view {
    const char* data_; // Pointer to the start of the character sequence
    size_t size_;      // The length of the sequence
};

  • Why you should use it!

    1. Avoids Unnecessary Allocations : This is the most common use case. Consider a function that takes a string argument.

      void foo(const std::string& s) {
    // ... logic
}

int main() {
    foo("Hello World"); 
}

      When you pass the literal "Hello World" (which is a const char*) to a function expecting const std::string&, the compiler secretly creates a temporary std::string. This involves allocating memory on the heap, copying the characters, and then freeing the memory when the function returns.

      To avoid these overheads, do

      void foo(std::string_view s) {
    // ... logic
}

int main() {
    foo("Hello World");
}

      No std::string is created. The string_view simply points to the static memory where "Hello World" lives.

    2. Cheaper slicing of strings : If you want to process, use a substring of a string, std::string::substr creates a brand new string. std::string_view::substr does not copy. It simply adjusts the internal pointer and size variables.

    3. Good software dev practice enforcing the DRY principle : Before std::string_view it was very common to see two versions of a function, one for const char* and other for std::string. std::string_view simplifies this redundancy.

  • Is it all flowers?

    • Unfortunately, the issue regarding the lifetime still persists. We must ensure that the data std::string_view points to outlives its own lifetime to guarantee that the program is well defined. For instance something like this is entering dangerous territories
      std::string_view get_bad_view() {
    std::string s = "Hello";
    return s;
}

2. std::span<T>

Just like std::string_view, std::span is incredibly lightweight. It is a "fat pointer", it carries the memory address and the number of elements. Internally, it looks roughly like this:

template <typename T>
class span {
    T* data_;      
    size_t size_;  
};
  • Why you should use it!

    1. Fixes the pointer decay problem : Passing raw array to a function causes it to lose it's information about size. So every function where we are passing arrays (usually) has an extra argument to specify the size of the array. Now size is implicitly present

      Before std::span<T>

      void foo(int* arr, size_t size) { 
    // ... logic
}

int arr[10];
foo(arr, 10);

      After std::span<T>

      void foo(std::span<int> data) {
    // ... logic
}

int arr[10];
std::vector<int> vec = {1, 2, 3};

foo(arr); 
foo(vec);

    2. Slicing again : Just like std::string_view::substr, span has subspan. You can pass a "window" of a vector to a function without copying the elements.

      void foo(std::span<int> s) {
    // .. do something with the subarray
}

std::vector<int> numbers = {10, 5, 8, 2, 1, 9};
foo(std::span(numbers).subspan(2, 3));

    3. (Potential) Compile time optimizations : std::span has a second template parameter: std::span<T, Extent>.

      • std::dynamic_extent (Default): Stores pointer + size (2 words).
      • N (Fixed size): Stores only the pointer (1 word). The size is hardcoded in the type.

      If you know the size of your data at compile time, std::span<int, N> is slightly more efficient than std::span<int> due to compiler optimization techniques.

    4. Byte manipulation : std::span allows you to view any object as a sequence of raw bytes (std::byte or unsigned char) in a safe fashion without performing manual pointer arithmetic and "unsafe" casts.

      Before std::span

      struct Bar {
    uint16_t id;
    uint32_t len;
};

void send_data(const char* bytes, size_t length);

Bar b {1, 1024};

// Danger : We manually calculate the size, what if we intend to send only id? Need to do some calculations
send_data(reinterpret_cast<const char*>(&b), sizeof(b));

      After std::span, std::as_bytes and std::as_writable_bytes take a span of any type and convert it into a span of const std::byte (or std::byte).

      #include <span>
#include <cstddef> 

struct Bar {
    uint16_t id;
    uint32_t len;
};

void send_data(std::span<const std::byte> bytes);

Bar b {1, 1024};

send_data(std::as_bytes(std::span{&b, 1}));

      Why is it better?

      • Automatic bounds : The resulting std::span<std::byte> automatically has the size sizeof(T) * original_span_size. You cannot accidentally claim the buffer is larger or smaller than it actually is.
      • Const correctness : std::as_bytesalways returns std::span<const std::byte>. You cannot accidentally overwrite the struct data while trying to read it.

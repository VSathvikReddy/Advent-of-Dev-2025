Tail Recursion

Consider a recursive function that calculates the factorial of a number, modulo 1e9+7.

def factorial(n):
    if n == 0:
        return 1
    return (n * factorial(n - 1))%1000000007

Try calling factorial(5). Works, right? How about factorial(100)? Still works? Try factorial(998). Still works?? I didn't think so. This is because python has a stack limit of 999, and when you do factorial(998) the stack will try to extend till 1000.

Now, let's try rewriting the function a little bit.

def factorial_but_better(n, acc):
    if n==0:
        return acc
    return factorial_but_better(n-1, (n*acc)%1000000007)

First, understand what is going on, and convince yourself that factorial_but_better(n,1) does the same thing as factorial(n). Now, try calling factorial_but_better(998, 1). It doesn't work? No shit, python is stupid. Try writing the same code in cpp. Note that cpp doesn't set a limit on recursive calls, but instead sets a limit on the size of the stack, so to get a segfault on the factorial function, you may have to go until factorial(700000) or so. Use binary search to find the limit.

So what's going on?

To see what the difference is, look at what each function saves on the stack, and what computation is left to do after the recursive call. "factorial" has to do a multiplication and a modulo after the call, but "factorial_but_better" has nothing left to do!.

This is what we call tail recursion. A tail recursive function, simply put, is a function that does not have any more computation to do after any recursive call. Why does this help us?

Tail Call Optimization

When you have a tail recursive function, there is no computation to do after the recursive call. This implies that after the recursive call, the stack frame of the caller function is completely useless. Most modern compilers take advantage of this fact, and reuse the same frame for the next recursive call. This is called Tail Call Optimization.

This allows the compiler to complete the entire recursion in a single stack frame. You may be familiar with how most recursive functions have an iterative counterpart that runs faster. TCO is the compiler optimization that converts a tail recursive call to its iterative counterpart.

To understand the recursion to iteration concept better, think about how you would go about converting a recursive function to an iterative function. Specifically think about how you would do it for a non tail recursive function and for a tail recursive function. You will notice that converting a tail recursive function to a loop is trivial, and converting a non tail recursive function to a tail recursive function is basically the same as loopifying it.

If you are still wondering, python didn't improve with tail recursion because the python interpreter does not support tail call optimization.

Continuation Passing Style

CPS is a very interesting style of coding, primarily seen in functional programming, where all functions are pure, have no side effects, and whose only purpose is to calculate a result value and return it. The core idea is that control is passed explicitly in the form of a continuation.

All functions take in an extra argument, the continuation. When the function computes its result value, it returns it by calling the continuation function with this value as the argument.

It can get a little tricky to implement in cpp, so for all the remaining problems, a pseudo-code answer in style of the example below will be permitted.

def factorial_CPS(n, k : int->_):
    if(n==0):
        k(1)
    else:
        factorial_CPS (n-1, (r => k(n*r)))

The above is an example of the factorial function using CPS. Read k : int->int as k, a function that takes an integer and returns whatever it wants. Read (r => k(n*r)) a function that takes a value r as parameter, and performs k(n*r). To call the function we would pass the identity function as the second argument initially.

In C++ you would need to use lambdas and function pointers to implement this.

To understand why this function works, try dry-running factorial_CPS(3, (r => f(r)))).
This function would then call factorial_CPS(2, (r => f(3*r))).
This would in turn call factorial_CPS(1, (r => f(2*3*r))).
This would in turn call factorial_CPS(0, (r => f(1*2*3*r).
This would in turn call f(1*2*3*1).
In this way, the factorial has been calculated, and the continuation, which must have been encoded in f, will be called with the factorial as its argument.

Spend some time in understanding this by trying to write other recursive functions in CPS, and then return to Problem 3, the Ackermann function.

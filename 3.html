Compiler optimizations

A compiler optimization is a transformation performed by the compiler to make your program faster, smaller, or more efficient, without changing its observable behavior.

In practice, the compiler may:

  • eliminate unnecessary computations (constant folding, dead-code elimination)

  • replace expressions with their precomputed values (constant propagation)

  • remove or merge temporary objects

  • inline small functions

  • unroll or reorder loops

  • reorder instructions for better CPU performance

  • optimize away entire blocks of code proven redundant

We will now look at one such mechanism that helps shift work from runtime to compile time and reduce overhead: constexpr.

constexpr

This is a keyword introduced in C++ 11 which allows certain expressions, variables, and functions to be evaluated at compile-time rather than at run time.

Programming using constexpr is usually called Compile-time programming for this very reason.

At its core, constexpr tells the compiler, "This value or function may be evaluated at compile time if all its inputs are known at compile time."

This keyword doesn't force compile-time evaluation, it simply just permits it. Hence it need not always happen.

Why constexpr exists

Many values in a program are naturally constant. Examples:

  • mathematical calculations

  • precomputed lookup tables

  • fixed algorithmic results (say, factorial(10))

  • configuration constants

  • operations on literal data

Before constexpr, such compile-time computations required macros, template metaprogramming, or manual precomputation. Modern C++ provides a clean, safe, expressive way to compute these values automatically at compile time.

Where can constexpr be used?

constexpr variables 

constexpr int x = 10;
constexpr int y = a; // here a must be a compile-time constant else it'll throw an error

A constexpr variable must be initialised using a compile-time constant expression.

If any value which is not a constant expression is used, it'll throw an error.

Another useful construct to learn in compile-time programming is static_assert(bool).

Throws an error if the value inside is false and throws a different error if the expression inside contains a value which is not a compile-time constant.

constexpr int x = 10;
int y = 10;
assert(x == y); // works fine, rather would only be evaluated during run time
static_assert(x == y); // throws error during the compilation phase

constexpr functions

A constexpr function is a function that would be computed during compilation if

  1. The arguments are compile-time constants.
  2. The function must be capable of constant evaluation (no side effects on the compile-time path).

Time for another side track :)

In programming, a side effect refers to any observable change to the state of a program or its environment that occurs as a result of executing a function or expression, beyond merely returning a value. Essentially, it's an action that a piece of code performs that affects something outside of its immediate scope.

Examples of side effects include:

  • Modifying a non-local variable: Changing the value of a global variable or a variable in an outer scope.
  • Performing I/O operations: Reading from or writing to a file, database, network, or console (like print statements).
  • Mutating mutable data structures: Modifying an object or array passed by reference.

There are many other examples of side effects but we won't really concern ourselves with that for now.

Back to constexpr, look at the following example

constexpr int f(int x) {
    return x+1;
}

int main() {
    constexpr int a = f(1); // evaluated during compile time
    int b = f(2); // evaluated during runtime
    int n;
    cin >> n;
    constexpr int c = f(n); // compile time error, a constexpr depends on a non compile-time constant
    int d = f(n); // evaluated during runtime
}

A keyword we can use to make sure the function in only evaluated during compile time is consteval.

It is a keyword that was introduced in C++20.

consteval int f(int x) {
    return x+1;
}

int main() {
    constexpr int a = f(1); // evaluated during compile time
    int b = f(2); // IS EVALUATED AT COMPILE TIME NOW!!!
    int n;
    cin >> n;
    constexpr int c = f(n); // compile time error, a constexpr depends on a non compile-time constant
    int d = f(n); // compile time error
}

Note: consteval forces a function call to be evaluated at compile time, so calls like f(2) are evaluated during compilation if 2 is a constant expression, even when the result is assigned to a runtime variable. By contrast, a constexpr function may be evaluated at compile time, but calls in ordinary runtime contexts (like int b = f(2);) are not required to be, the compiler may choose to evaluate them at runtime.

In short: consteval mandates compile-time evaluation (when possible), constexpr permits it.

Another side track

Till now, you've either just taken my word for something being evaluated during compile time or run time, or you've just been sceptical the entire time.

Don't worry I've got something for you ;)

Introducing std::is_constant_evaluated().

This construct returns true if it is in compilation phase and returns false if it is in run time phase.

So lets say we replace our function f to the following for experimental purposes:

constexpr int f(int x) {
    if(std::is_constant_evaluated()) {
        return 2*x;
    }
    return x;
}

int main() {
    constexpr int a = f(1); // evaluated during compile time
    int b = f(2); // evaluated during runtime
    int n;
    cin >> n; // say I input 3
    // constexpr int c = f(n); // compile time error, a constexpr depends on a non compile-time constant
    int d = f(n); // evaluated during runtime
    std::cout << a << " " << b << " " << d << std::endl;
}

Output:

2 2 3

Same thing if I replace the constexpr with consteval, the output is (and comment out the d part):

2 4

Btw I found this (int b = f(2)'s behaviour, I thought it should've been an error) out as I was making this example at 1:00 am in the night, so messing around and finding out works very well and I highly encourage you to find more interesting behaviour :)

constexpr int f(int x) {
    if(std::is_constant_evaluated()) {
        // The next comment, if uncommented would lead to an error
        // std::cout << "compile time " << x << std::endl; // I/O is not allowed in the compile-time control flow
        return x;
    }
    std::cout << "run time " << x << std::endl; // control flow can never lead to this during compile time so its fine
    return x;
}

constexpr objects

Just a small example for this.

struct Point {
    int x, y;
    constexpr Point(int a, int b) : x(a), y(b) {}
};

constexpr Point p(3, 4); // created at compile time

Please do note that all the explanation above is based on the following compilation scheme:

g++ -std=c++20 test.cpp

Enabling other optimizations might change the behaviour of the given snippets above but the core concept remains the same :)

Keywords used with Constructors

= delete 

struct A {
    int x;
    A(int) = delete;
};

What does this do?

  • This tells the compiler that: If anyone uses the int constructor for A, throw a compile-time error. But do note that it need not be always used with a constructor.

  • So basically, it marks a function as deleted. The function participates in overload resolution, but if selected the program is ill-formed (compile error).

  • Commonly used to forbid certain constructions or implicit conversions.

Why use this?

  • Prevent compiler defaults or even inherited functions
class A {
public:
    A(const A&) = delete; // disable the copy constructor
    A& operator= (const A&) = delete; // disable the copy assignment
    void* operator new(size_t) = delete; // now we can't have heap allocated A objects

    void print() {cout << "Hehehehaw" << endl;}
private:
    int* x;
};

class B: public A {
public:
    void print() = delete; // now we can't call print from a B object.
};
  • Prevent conversions from some types
class C {
public:
    C(const string& _s): s(_s) {}
    C(char*) = delete; // without this you can initialise this object with char* but now you can't
private:
    string s;
};

int main () {
    string str;
    char* c_str;

}
  • Remove unnecessary overloads
class D {
public:
    template<typename T>
    void throwIntoBasement(T& t) {/* throw into basement :)*/}

    void throwIntoBasement(bool) = delete; // I don't want bools in my basement
};

int main() {
    D d;
    int k = 10;
    d.throwIntoBasement(k); // compiles
    bool l = true;
    d.throwIntoBasement(l); // compilation error
}

= default

What does this do?

  • First off, unlike delete, this can't be used with any method. It can only be used by a few special methods.

  • These special methods are: Default constructor, Destructor, Copy constructor, Copy assignment operator, Move constructor and Move assignment operator.

  • It requests that the compiler generate the default implementation for a special member.

Why use this?

  • Re-enable a special member you would otherwise lose.
class E {
public:
    E(int _tempo) : tempo(_tempo) {} // defining one constructor gets rid of the default constructor
    E() = default; // enables the default constructor
private:
    int tempo;
};

int main() {
    E(120);
    E(); // would've thrown an error without the default constructor
}
  • Make intent explicit for readability and overload resolution (even delete helps to do the same).

A few facts

  • A defaulted method is implicitly inline.

  • A defaulted method can be noexcept as well. F(F&&) noexcept = default;

  • You can define the method inside and default it outside as follows, but now the method i not implicitly inlined anymore.

class G{
public:
    G();
};

G::G() = default;

explicit

What does this do?

  • It applies to constructors or conversion operations to forbid implicit conversions

  • In C++20 we can also write explicit(bool) to conditionally make a constructor/conversion explicit.

Why use this?

  • To avoid unintended implicit conversion.

  • Make the object construction clearer by prefering H h{val}; or H h(val); over H h = val;

class I {
public:
    explicit I(int x) {
        var = x;
    } 
private:
    int var;
};

int main() {
    I i(10); // this is fine
    I o = 10; // error, would work without explicit
}

Welcome back to the Side Track :)

So you might wonder why I o = 10 would work if not for the explicit keyword.

What happens is that I o = 10 is converted to I o(10). What explicit does here is to block this conversion, rather throw a compilation error.

This is only possible for non explicit constructors that take only one paramater. Such constructors are called Conversion Constructors.

Please do note that the method can take one parameter if the other parameters have default values.

Let us have a look at the conditionally explicit constructors

template<typename K>
struct J {
    explicit(!std::is_convertible_v<K, int>)
    J(K value) {}
};

int main() {
    J<int> o1 = 10; // fine
    J<string> o2 = string{"hi"}; // implicit construction error
}

Another side track, I found something interesting as I was researching for this :)

For a class/struct, we have an operator called operator bool().

What this basically does it tells the compiler that your object can be treated as a boolean value as well.

struct L {
    operator bool() {return true;}
};

Now here's the problem.

int main() {
    L l;
    int x = l; // this will work and assign x the value of 1 from the bool operator
}

If the above behaviour is unintended, which in most cases would be true, we can use explicit to fix it :)

struct L {
    explicit operator bool() {return true;}
};

int main() {
    L l;
    if(l) cout << "hello :)\n"; // this would work as intended.
    int x = l; // implicit construction error.
}

constexpr

  • Our good old friend :), I hope I don't have to elaborate on this one.

  • Brief description: constexpr tries evaluating the object at compile time if possible. This helps us make compile time objects!!

  • constexpr constructors can also be used at runtime.

noexcept

What does this do?

  • It is a promise to the compiler that says, "Trust me, I won't throw an error."

  • This can also be conditional similar to explicit.

Why use this?

  • Enables optimizations thanks to exception safety guarantees, but if it does throw an exception or have an error, the program would terminate.

  • Standard library prefers copy over move constructor unless your defined move constructor uses noexcept.

Aggregates

  • An aggregate is a simple class or struct whose members can be initialized directly using braces without needing a constructor.

  • Aggregates behave like "plain data bags".

  • C++ allows you to initialize them using aggregate initialization, which directly initializes each member in order of appearance.

When is a struct/class an aggregate?

This is from Stack Overflow 10 years ago :)

A class is an aggregate if:

  • All of its data members are public
  • It does not define any constructors
  • It has no in-class initializers
  • It has no base classes or virtual functions
In Modern C++:

A class/struct is an aggregate if it has no:

  • User-provided or user-declared constructors.
  • Private or protected non-static data members.
  • Virtual functions.
  • In-class initializers for non-static data members (relaxed in C++14 and relaxed further in C++20).
  • Base classes (relaxed in C++17 to allow public, non-virtual aggregate base classes).
struct M {
    int x;
    int y;
};

M p{10, 20}; // aggregate initialization

struct N {
    int a;
    double b = 3.14;  // still an aggregate
};

N n{1};  // a = 1, b = 3.14

int arr[3] = {1,2,3}; // arrays are also aggregates.

class O {
public:
    int a;
    double b;
};

class P : public O {
public:
    int c;
    O d;
    vector<int> e;
};

P obj{{1,2.0},4,{5,6.0},{7,8,9}};

Why do aggregates matter?

  • They allow brace initialization, which is clean and safe.
  • They guarantee member order–based initialization, which is predictable.
  • They eliminate constructor overhead.
  • They’re widely used in:
    • simple data transfer objects (DTOs)
    • compile-time structs
    • POD-style types (plain old data)

Fun fact :) 

struct Q {
    Q(int a) {
        this->a = a;
    }
    int a;
};

Q q{3};

You might think the above code will result in an error, but it won't :)

Many people initially assume that:

  • braces {} → only work for aggregates
  • if a type has a constructor, braces shouldn’t work

But that’s not true.

Just because a struct/class is not an aggregate does not mean you can't use brace initialization.

What happens with Q q{3} then?

First the compiler checks whether Q is an aggregate.

  • If Q is an aggregate -> aggregate initialization
  • If Q is not an aggregate -> constructor overload resolution happens instead

So it is technically interpreted as Q q(3) instead.
